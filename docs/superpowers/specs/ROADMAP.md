## 2. Вкладка «Telegram-пользователи» + блокировка 🟡 частично

**✅ Шаг 1 — блокировка пользователя (quick-win)** — РЕАЛИЗОВАНО и влито в main
(2026-06-13, merge `c3121ba`). Спец `specs/2026-06-13-user-blocking-design.md`,
план `plans/2026-06-13-user-blocking-plan.md`. `users.is_blocked`/`blocked_at`,
JWT-клейм `src`, гард в `get_current_user_id` (telegram-сессии → 403
`user_blocked`, немедленно), отказ на `/auth/telegram`, admin block/unblock на
странице «Пользователи». Логин/пароль не затрагивается.

**⬜ Шаг 2 — полная вкладка «Telegram-пользователи»** (не спроектирован).
Запрос: отдельная вкладка со ВСЕМИ TG-пользователями и полной инфой —
аватарки, телефоны, username, имя; статус «является ли клиентом»; кликабельные
юзернеймы (`t.me/<username>`).

**⚠️ Ограничения Telegram, которые надо решить ДО проектирования:**
- **Телефон** Telegram не отдаёт автоматически — только через явный
  `requestContact` (бот) / `requestContact` (Mini App). Нужен флоу сбора; иначе
  колонка пустая. Решить: запрашивать контакт при первом входе? делать
  опциональным?
- **Аватар** — частично: `getUserProfilePhotos(user_id)` (если не закрыт
  приватностью) или `photo_url` из initData. Нужен фоновый сбор/кеш.
- **«Клиент»** — определить: у кого есть сделка/долг? кто в справочнике
  контрагентов? Влияет на схему.

Блокировку (маленькую и чистую) можно вынести вперёд как quick-win:
`users.is_blocked` + гард в **initData-пути** аутентификации (403), а
логин/пароль-путь блок игнорирует. Покрыть тестом: заблокированный не входит
через initData, но входит по паролю.
