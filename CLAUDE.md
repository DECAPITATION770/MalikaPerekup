# CLAUDE.md

Поведенческие правила для LLM-агентов в этом репозитории.

**Tradeoff:** правила склоняют к осторожности, а не к скорости. Для тривиальных задач — здравый смысл.

## 1. Думай, прежде чем кодить

**Не предполагай. Не прячь сомнения. Озвучивай tradeoff'ы.**

- Перед реализацией — явно сформулируй допущения. Если не уверен — спроси.
- Если возможны несколько интерпретаций — покажи их, не выбирай молча.
- Если есть более простой путь — скажи. Возражай, если по делу.
- Если что-то неясно — стоп. Назови, что мешает. Спроси.

## 2. Минимум кода

- Никаких фич сверх запрошенного.
- Никаких абстракций для одноразового кода.
- Никакой «гибкости» или «конфигурируемости», которую не просили.
- Никакой обработки ошибок для невозможных сценариев.
- Если 200 строк сделали 50 — перепиши.

Тест: «Сениор сказал бы, что это переусложнено?» Если да — упрости.

## 3. Хирургические изменения

- Не «улучшай» соседний код, комментарии, форматирование.
- Не рефактори то, что не сломано.
- Соблюдай существующий стиль, даже если сам делал бы иначе.
- Заметил неиспользуемое — упомяни, не удаляй.

Тест: каждая изменённая строка должна напрямую следовать из запроса пользователя.

## 4. Цель-ориентированное исполнение

Превращай задачу в проверяемую цель:
- «Добавь валидацию» → «Напиши тесты на невалидный ввод, потом сделай чтобы прошли»
- «Почини баг» → «Напиши тест воспроизводящий, потом сделай чтобы прошёл»
- «Отрефактори X» → «Тесты зелёные до и после»

Для многошаговых — короткий план:
```
1. [шаг] → проверка: [что]
2. [шаг] → проверка: [что]
```

## 5. Стек — не тащить лишнее

- **Backend:** Python 3.12, FastAPI, aiogram 3.x, SQLAlchemy 2.0, Alembic, Postgres 16, Redis. Менеджер — `uv`.
- **Frontend (tenant):** React 18 + Vite + TS, `@telegram-apps/sdk-react`, Tailwind, TanStack Query, react-hook-form + zod, react-i18next. Менеджер — `pnpm`.
- **Object storage:** MinIO локально / Cloudflare R2 в проде.
- Новые библиотеки — только после обсуждения. Сначала проверь, нет ли в стеке.

## 6. Multi-tenant SaaS — критическая инвариантность

Malika v2 — multi-tenant SaaS по подписочной модели. Каждый магазин = один `Tenant`. `Tenant`-ы создаёт super-admin через `/api/admin/tenants` (UI в Mini App).

**Каждый запрос к бизнес-данным ОБЯЗАН фильтроваться по `tenant_id` текущего пользователя.**

- Использовать FastAPI dependency `current_user` (выдаёт `User` + `tenant_id` из JWT) + явный фильтр `WHERE tenant_id = current_user.tenant_id` в репозиториях.
- Никаких сырых SELECT/UPDATE/DELETE без `tenant_id`-фильтра.
- Запрещено принимать `tenant_id` из тела запроса или query-параметра — только из аутентифицированного JWT.
- Super-admin (`role='super_admin'`, `tenant_id=NULL`) — единственная роль с доступом к данным разных tenant'ов, только через `/api/admin/*` роуты, **не через `/api/*`** обычные.
- В тестах обязательно сценарий: пользователь tenant A не видит данные tenant B (см. §13).

Это сознательный реверс v2-roadmap от 2026-05-28: изначально v2 планировался single-tenant, но бизнес-модель «подписка магазинам» требует multi-tenant. Решение зафиксировано в [[v2-roadmap]] секции «История архитектурных реверсов».

## 7. Авторизация — Telegram + резервный логин/пароль

Два способа входа в один аккаунт:

1. **Telegram (основной):** Mini App шлёт `initData` → верификация HMAC-SHA256 от `BOT_TOKEN`. JWT на 24 часа.
2. **Логин/пароль (резервный):** настраивается в Settings, для входа когда Telegram недоступен. POST `/auth/login` → JWT с тем же `user_id`.

**Правила:**
- Никогда не доверять `tg_id`/`user_id` из тела запроса — только из верифицированного источника (initData или валидный JWT).
- Пароль — `bcrypt`, cost ≥ 12. Никогда не plaintext.
- Логин/пароль — опциональные поля у пользователя (`NULL`, если не настраивал).
- В dev-режиме bypass только при явном `DEV_AUTH_BYPASS=true` в `.env`. В проде `app/main.py:startup()` падает, если он включён.

## 8. Деньги

- Все суммы (UZS, USD) — `Decimal` в Python и `string` в API. Никогда `float`.
- В БД — `NUMERIC(18, 2)` для UZS, `NUMERIC(14, 2)` для USD.
- Курс валюты на момент сделки — фиксируется в самой сделке, не пересчитывается задним числом.
- Расчёт прибыли (`sale_price - purchase_price`) — в одном месте, покрыт тестами.

## 9. PII (паспорта, фото документов)

- Фото документов продавца/покупателя — PII. Хранить в MinIO/R2 с **private ACL**.
- Раздавать только через подписанные URL с TTL ≤ 15 минут.
- Никогда не логировать содержимое `*_doc_number`, `*_phone`, `seller_photos`, `buyer_photos`.
- При удалении сделки — каскадно удалять файлы из object storage.

## 10. Domain events (v2 эксперимент)

Эта гипотеза из v2-roadmap пункт #5. Реализуется в Stage D — описано в roadmap.

- Каждое изменение состояния — append-only event (`purchase_recorded`, `device_sold`, `installment_paid`).
- Текущее состояние — projection из event log.
- Undo = новый event (`*_reversed`), не DELETE.

Если на практике окажется overkill — откатываемся к обычным mutable models. Решение задокументировать в roadmap.

## 11. i18n (RU + UZ-Latn)

- Никаких хардкод-строк в UI и в сообщениях бота. Только ключи + словари.
- Frontend — `react-i18next`, `tenant/src/i18n/{ru,uz}.json`.
- Backend (бот, ошибки API) — словари в `backend/app/i18n/{ru,uz}.json`, выбор по `user.language`.
- Дата/время — таймзона `Asia/Tashkent`. Хранить в UTC, отображать в локальной.

## 12. Миграции БД

- Любое изменение схемы — через Alembic.
- Не править существующую миграцию после мерджа — только новая ревизия.
- Перед коммитом: `alembic upgrade head` локально, затем `alembic downgrade -1 && alembic upgrade head` для проверки обратимости.

## 13. Тесты

- Минимум pytest-покрытие для критичного: `verify_telegram_initdata`, `profit_calc`, расчёт графика рассрочки, projection event → state, **изоляция tenant'ов** (tenant A не видит данные B — на каждую новую бизнес-фичу).
- Интеграционные тесты — против реального Postgres в Docker, не SQLite-моков.
- Перед PR — `cd backend && uv run pytest` зелёный.

## 15. UX-приоритеты

- **Тёмная тема по умолчанию** — Mini App наследует тему Telegram, но проектировать dark-first: рынок Малика, яркое солнце. Кнопки крупные, текст контрастный.
- **«Nasiya»** — в UZ-интерфейсе вместо «рассрочка». В RU — «рассрочка».
- **Без catalog feature** — модели не хранятся как справочник, а автокомплитятся из истории закупок (v2-roadmap пункт #4).
- **Дашборд: 3 ключевых числа** — Прибыль сегодня / Замороженные деньги (стоимость витрины) / Долги по Nasiya.

## 16. Что НЕ делать в v2

- Не поднимать catalog feature без явного обсуждения.
- Не интегрировать эквайринг (Click/Payme) до запроса — все сделки cash.
- Не добавлять полноценную бухгалтерию (НДС/КПН).
- Не использовать SMS — все уведомления через Telegram.
- Не строить фичу, пока не сформулирована success criteria для неё.
- Не давать super-admin прямой доступ к бизнес-данным конкретного tenant'а (всё через провижининг/администрирование, не через «войти как»). Если потребуется debug — отдельный механизм impersonate с явным аудит-логом.
